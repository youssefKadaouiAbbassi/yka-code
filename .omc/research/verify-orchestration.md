## TASK 3 — COMPOSITION / MUTUAL EXCLUSIVITY MAP

### 3a — Platforms that compose cleanly

| Pair | Reason |
|---|---|
| **Paperclip + Vibe-Kanban** | Paperclip owns the company/HR abstraction; Vibe-Kanban owns the coding-agent execution layer. Different layers, no overlap. |
| **Gastown + ComposioHQ/agent-orchestrator** | Gastown handles worktree orchestration and merge queue; Composio handles CI healing and PR creation. Both can share the same git worktrees. |
| **claude-squad + multica** | claude-squad is a terminal tmux manager; multica is the Kanban/dashboard layer. Squad handles runtime, Multica handles planning UI. |
| **oh-my-claudecode + any harness** | OMC is a plugin/skill/hooks layer that sits on top of any Claude Code instance; it doesn't own the session loop. Composes with Paperclip, Gastown, Vibe-Kanban workers. |
| **Superpowers + swarm tools** | Superpowers is prompt-level discipline only (no harness enforcement); any of the worktree-spawning tools can run Superpowers-configured agents as workers. |
| **metaswarm + ralph-orchestrator** | metaswarm provides agent specialization; ralph provides the loop-runner pattern. Different concerns. |
| **sandstorm + any orchestrator** | Sandstorm is a sandbox execution backend; any orchestrator can delegate to it for isolated execution. |
| **praktor / ductor + any task orchestrator** | Both are I/O front-ends (Telegram gateway); they don't own the orchestration logic, so compose with any backend. |
| **bernstein + vibe-kanban** | Bernstein handles declarative YAML pipeline definition; Vibe-Kanban handles the execution UI. Different layers. |
| **moonrunnerkc/swarm-orchestrator + any orchestrator** | Pure governance/quality-gate layer; designed to wrap other orchestrators. |

### 3b — Mutually exclusive (both wrap the session loop / task queue)

| Conflict Pair | Reason |
|---|---|
| **Paperclip vs Gastown** | Both own the top-level task dispatch and worker lifecycle. Running both would create two competing schedulers. |
| **Paperclip vs multica** | Both provide the "assign issue to agent + track progress" abstraction. Overlapping Kanban/dashboard ownership. |
| **Gastown vs ComposioHQ/agent-orchestrator** | Both spawn agents in worktrees and manage merge queues. Use one or the other for the same repo. |
| **claude-squad vs multiclaude** | Both manage multiple Claude sessions in tmux with git-worktree isolation; architecturally identical layer. |
| **ruflo vs multiclaude** | Ruflo's multi-agent mode and multiclaude both own the session-spawn + merge-queue loop. |
| **vibe-kanban vs multica vs Claw-Kanban** | All three are Kanban boards that assign tasks to agents. Running two for the same project creates conflicting task state. |
| **ralph-orchestrator vs oh-my-claudecode ralph skill** | Both implement the Ralph loop pattern. The OMC `ralph` skill and `mikeyobrien/ralph-orchestrator` solve the same problem; pick one. |
| **praktor vs ductor** | Both are Telegram-gateway + Docker-sandboxed agent runners. Near-identical architecture; pick one. |

### 3c — Platforms potentially obsoleted by Claude Code native Agent Teams

Claude Code native Agent Teams (enable via `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`) provides: team lead + teammates, shared task list, `SendMessage`, file-locked task claims.

| Platform | Obsoleted? | Assessment |
